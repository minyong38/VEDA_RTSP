# Design Decisions

> 면접에서 받을 만한 질문에 답하기 위한 기록. 결정의 *이유*를 남긴다.

## Q. 왜 RTSP는 직접 구현하고 RTP는 라이브러리를 쓰나?

**RTSP 시그널링이 프로토콜 이해의 핵심이기 때문.**

- RTSP는 텍스트 기반 프로토콜로, 요청/응답 구조와 상태머신을 직접 구현하면서 프로토콜 설계 사상을 깊이 이해할 수 있다.
- RTP는 바이너리 패킷화로, 이미 검증된 라이브러리가 많고 직접 구현 대비 학습 가치가 상대적으로 낮다.
- 시간 효율: RTSP 시그널링에 집중하고 RTP는 라이브러리로 안정성 확보.

**포트폴리오 관점:**
- "RTSP를 RFC 문서 보고 직접 구현했다"는 네트워크 프로토콜 이해도를 증명.
- RTP 라이브러리 연동 경험은 시스템 통합 능력을 보여줌.

## Q. 왜 live555의 RTSP 서버는 안 쓰나?

**"RTSP 직접 구현"의 의미가 없어지기 때문.**

live555는 RTSP + RTP를 모두 처리하는 완성된 라이브러리다.
이걸 쓰면 편하지만, 면접에서 "RTSP 프로토콜이 어떻게 동작하나요?"에 대답하기 어렵다.

→ live555의 **RTP 전송 부분만** 활용하고, RTSP 시그널링은 직접 구현한다.

## Q. RTSP는 TCP, RTP는 UDP — 왜 다른 전송?

- **RTSP (제어)**: 요청-응답이 정확히 도착해야 함 → TCP
- **RTP (영상 데이터)**: 약간의 손실은 허용, 지연이 더 치명적 → UDP
  - TCP의 재전송 대기 > 다음 프레임 도착이면 의미 없음

## Q. C++로 짠 이유는?

- POSIX 소켓 API와 자연스럽게 어울림
- 메모리 레이아웃 직접 제어 가능
- 임베디드/카메라 펌웨어 분야의 주력 언어

## Q. SDP 생성 시 주의점은?

- `profile-level-id`: 실제 인코더 설정과 일치해야 VLC가 디코더 준비 가능
- `packetization-mode=1`: FU-A 분할 사용 시 필수
- `a=control:track1`: SETUP 요청에서 트랙 식별에 사용
